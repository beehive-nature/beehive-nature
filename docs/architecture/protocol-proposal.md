# Zano Hardware-Wallet Signing Protocol — Design Proposal (DRAFT v0.1)

**Purpose:** a concrete, reviewable proposal to add native ZANO support to Trezor Core firmware, to be opened with **SatoshiLabs** (device/firmware) and the **Zano core team** (host wallet + cryptography). Modeled directly on the Monero-Trezor cold-wallet integration (Klinec & Matyáš; ph4r05/monero-trezor-doc), which is the closest existing precedent.

Companion artifact: `messages-zano.proto` (the wire format this document describes).

---

## 1. Objective & non-goals

**Objective (v1):** sign transfers of **native ZANO** on-device, with spend keys never leaving the Trezor, driven from the Zano desktop wallet acting as an untrusted coordinator ("agent"). Device behaves as a cold wallet: it builds the whole transaction incrementally and releases signatures only on a correct protocol run.

**Non-goals for v1 (wire-reserved, not implemented):**
- Confidential Assets send/receive (native-only in v1; asset fields exist in the proto for forward-compat).
- Zarcanum **staking** on-device (spending Zarcanum-era outputs is in scope; producing PoS mining txs is not).
- HF6 **Gateway Addresses** (account-based) and on-device bridging.
- Aliases as anything other than a display string; multisig/escrow; auditable-wallet issuance.
- Trezor Suite UI (v1 is driven from the Zano wallet, exactly as Monero is driven from Monero GUI).

Guarding this line is deliberate — Confidential Assets roughly double the crypto and UI surface.

---

## 2. Actors & trust model

- **Device (Trezor):** holds the seed and derives spend/view keys. Trusted. Constant, small RAM budget. Has a trusted display for user confirmation.
- **Agent (Zano wallet host):** knows the blockchain, selects decoys/ring members, stores offloaded state, performs offloaded heavy computation. **Untrusted** for integrity of what the user signs.
- **Network/daemon:** untrusted.

**Adversary:** an honest-but-curious *and* a malicious host that may reorder, replay, drop, duplicate, or tamper with messages, and interleave multiple protocol instances. The device must never (a) sign a transaction that differs from what the user confirmed on its screen, (b) leak a spend key or a usable partial signature out of order, or (c) be coerced into double-authorizing an input.

---

## 3. Security goals & mechanisms

1. **Cold-wallet equivalence.** The full tx is assembled on-device incrementally; the device's security properties equal Monero's analyzed cold-wallet protocol. Reusing this structure keeps the security argument tractable and eases eventual upstreaming into the Zano codebase.
2. **Sealed offload.** Because device RAM can't hold the tx, intermediate values (input descriptors, pseudo-out commitments, masks, spend keys) are handed to the host **HMAC-sealed with unique, index-dependent keys** derived per session. Tampering, reordering, replay, and reuse are all detected on return. This mirrors the Monero `vini_hmac` / `pseudo_out_hmac` pattern.
3. **Output pinning.** Output count and each destination are committed at `Init`; every later step is checked against the pinned set so the host cannot add/reorder/substitute outputs after user confirmation.
4. **Deferred signature release.** dv-CLSAG signatures are produced but kept sealed (`opening_key` in `FinalAck`) until a correct full run completes — a malicious host that aborts mid-protocol gains no usable signature.
5. **On-screen truth.** Recipient, amount, **asset (once CA lands)**, and fee are shown on the device's trusted display and are what actually gets signed.

---

## 4. Cryptographic surface (what differs from Monero)

Same Ed25519 group; same stealth-address and Pedersen-commitment machinery; Bulletproofs+ range proofs. The genuinely new device-side crypto:

- **dv-CLSAG signing.** The "different/variable" CLSAG variant lets a single ring mix explicit-asset and hidden-asset members. v1 (native-only) uses the explicit-asset path, but the signing routine and its transcript/domain-separation must match Zano's reference exactly — including the extra generator handling. `ring_asset_mode` in `SignInputRequest` selects the mode.
- **Zarcanum concealing point `Q`.** Post-HF4 confidential outputs carry an additional group element per output (`concealing_point` in `SetOutputAck`). Even non-staking transfers must construct and commit it.
- **Cross-asset balance proof.** In addition to per-output range proofs, Zano emits a balance proof tying commitments across asset types (`ZanoBalanceProofData`). v1 native-only still needs the native-asset balance relation to verify.

**Ground truth for all of the above is the Zano C++ reference implementation and the Cypher Stack–audited "Confidential Assets for RingCT and Zarcanum" paper — not blog posts.** Every generator, hash-to-point domain tag, and serialization order must be lifted from code and pinned with test vectors generated by the Zano node.

## 5. Key management & address derivation

- Derive Zano spend/view keys from the device seed. **Open decision:** follow Monero's precedent and use **SLIP-0010** (Trezor derives Monero keys via SLIP-0010 rather than the standard BIP-32/39 path, for the extra security the protocol needs). The known cost is that a SLIP-0010-derived backup may only be restorable to another Trezor, not to a stock Zano wallet.
  - **Ask for the Zano team:** is a Trezor-only recovery story acceptable, or must a device-derived wallet be importable into the standard Zano wallet? This choice drives the derivation scheme and must be settled before any code.
- Address types: `STANDARD`, `INTEGRATED` (payment id), `AUDITABLE`. `GATEWAY` reserved for post-HF6.
- Aliases are resolved host-side and shown on-device as an untrusted display hint only; routing is always by the raw address the user confirms.

---

## 6. Balance & key-image sync

Identical rationale to Monero: a view-only wallet that can't compute key images double-counts spent outputs and may pick already-spent inputs. The `ZanoKeyImageExportInit → Step* → Final` sub-protocol lets the device compute encrypted key images for owned outputs so the host maintains correct balances and valid input selection. Needed on first import / restore-from-device and after any wallet-cache loss.

---

## 7. Signing protocol (staged)

Ordered, stateful, mirroring Monero:

```
Init
  └─ pins: version, network, #outputs, destinations, change, fee, ring_size, hard_fork
  └─ returns: per-input HMAC keys (sealed session), rsig setup
(SetInput)*        one per input  → sealed vini / pseudo_out / masks / spend_key
(InputVini)*       replay inputs in key-image-sorted order → incremental prefix hash + committed permutation
AllInputsSet       → range-proof offload negotiation
(SetOutput)*       one per output → build output, Zarcanum Q, ecdh_info; verify offloaded BP+ chunk if >2 outputs
AllOutSet          → tx_prefix_hash, signature scaffold, balance_proof
(SignInput)*       one per input  → dv-CLSAG signature (kept sealed)
Final              → release opening_key + tx encryption keys
```

**Memory strategy:** device state stays **constant-size** across the run (independent of input count) — the Monero implementation demonstrates this and it is a hard design target here. Inputs and outputs stream through; nothing scales in device RAM with tx size.

---

## 8. Range-proof & balance-proof offload

- **≤2 outputs (the common case):** device computes the full Bulletproof+ itself.
- **>2 outputs:** heavy BP+ computation is offloaded to the host and the device **verifies** it, using the constant-device-memory MPC approach from the Monero work. `is_offloaded_bp` / `ZanoTransactionRsigData` carry the chunked proof material.
- The Zano-specific **balance proof** is produced by the device at `AllOutSet`; investigate whether any part can be safely offloaded/verified similarly, or must be fully on-device (likely fully on-device for v1 given native-only simplicity).

---

## 9. Wire protocol

See `messages-zano.proto`. Highlights vs. `messages-monero.proto`:
- Asset context threaded through source/destination/output/key-image structures (`asset_id`, and `[v2+]` `blinded_asset_id` / `asset_blinding_mask`).
- `concealing_point` (Zarcanum Q) on `SetOutputAck`.
- `ZanoBalanceProofData` on `AllOutSetAck` (no Monero analogue).
- `ring_asset_mode` on `SignInputRequest` to select the dv-CLSAG explicit/variable path.
- `ZanoAddressType` enum incl. reserved `GATEWAY`.

**Message IDs, `proto2`, package naming, and codegen annotations (`@start`/`@next`) follow Trezor conventions; IDs to be assigned by SatoshiLabs to avoid collisions.**

---

## 10. Serialization

Zano's transaction/prefix serialization differs from Monero's. A dedicated `apps/zano/serialize` module (analogue of `apps/monero/xmr/serialize`) ports Zano's exact byte layout from C++. The prefix hash the device signs must be byte-identical to what the Zano daemon validates — this is a common source of subtle, hard-to-debug failures and must be vector-tested early.

## 11. Open questions

**For the Zano core team**
1. Recovery/derivation: is SLIP-0010 (Trezor-only recovery) acceptable, or is standard-wallet import required?
2. Canonical source-of-truth references for dv-CLSAG, Zarcanum Q, and the balance proof (specific files/commits), plus a means to generate deterministic test vectors from the node.
3. Ownership of the host-side `device_zano` module in the Zano wallet repo, and release cadence.
4. Post-HF6 direction: should Gateway-Address signing be planned for v2, and does that change v1 wire choices?
5. Is there ecosystem-grant support for this work (Zano lists hardware-wallet integration on its roadmap)?

**For SatoshiLabs**
1. Implementation language for a new 2026 app on the Safe 7 — Rust-forward, or the MicroPython+C pattern the Monero app uses?
2. Message-ID allocation for the `zano` package.
3. Appetite/criteria for merging a new privacy-coin app (vs. Monero's history), and the required security-review path.
4. Any Safe 7 / Trezor Host Protocol (THP) specifics that affect a streaming, memory-bound signing protocol.
5. Reuse boundaries: which `trezor-crypto` Ed25519 / BP+ primitives can be shared with the Monero app vs. forked for Zano.

---

## 12. What we are asking for

- **Zano team:** crypto references + test vectors, agreement on derivation/recovery, and ownership of the host wallet module. Ideally, a named reviewer for the dv-CLSAG / balance-proof device implementation.
- **SatoshiLabs:** a design-review conversation, message-ID allocation, a language decision, and clarity on the merge + security-review bar.

**Deliverable sequencing** (from the parent plan): (1) crypto delta spec + off-device differential tests with node-generated vectors, (2) this protocol proposal + `messages-zano.proto`, (3) emulator `apps/zano/` skeleton signing a 1-in/2-out native-ZANO testnet tx end-to-end, (4) N-output offload + key-image sync, (5) host integration, hardening, external audit, upstream merge.
