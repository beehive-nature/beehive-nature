# Native ZANO Support on Trezor — Engineering Design & Delivery Plan

**Target device:** Trezor Safe 7 (and, for free, the rest of the Trezor Core family: Safe 5 / Safe 3 / Model T)
**Goal:** Native on-device custody + spend of native ZANO (not wZANO), with keys never leaving the device.
**Reference precedent:** Trezor's Monero (XMR) integration — same architectural class of problem.

---

## 0. Feasibility verdict

Feasible, but this is applied-cryptography firmware R&D, not a "add a coin definition" task. Zano sits in the same bucket as Monero, Cardano, Solana, and Tezos: coins that each required a **dedicated firmware app**, not a JSON entry. Realistic scope is a small specialist team over ~3–5 quarters, and the practical path to real users runs *through* SatoshiLabs (Trezor) and the Zano core team, not around them.

The reason it's hard is entirely the cryptography and the memory-constrained, multi-round signing protocol. Everything else (address display, derivation, client libraries) is well-trodden.

---

## 1. The Zano ⇄ Monero crypto delta (this is the crux)

The Trezor Monero app is the best starting scaffold, but Zano diverges in ways that require new on-device crypto:

| Element | Monero (existing on Trezor) | Zano (what you must build) |
|---|---|---|
| Curve | Ed25519 / Curve25519 | Same — good, primitives reusable |
| Ring signature | d-CLSAG | **d/v-CLSAG** ("different/variable") — supports mixing explicit and hidden asset amounts in one ring; new signing/verify math |
| Range proofs | Bulletproofs / BP+ | Bulletproofs+ — closest to Monero's current BP+, partly reusable |
| Assets per tx | Single asset (XMR) | **Confidential Assets**: multiple asset types per tx, blinded asset tags, per-asset generators, cross-asset balance proof |
| Consensus artifacts in tx | None on device | **Zarcanum ZPoS**: extra group element `Q` per confidential output; PoS mining tx shape (only relevant if you support staking) |
| Serialization | CryptoNote/Monero tx format | Zano tx format — needs its own serialize module |
| Extras | Payment IDs, subaddresses | Aliases, auditable wallets, escrow/multisig, (post-HF6) Gateway Addresses |

**Authoritative sources for the exact math** — do not implement from blog posts:
- Zano "Confidential Assets for RingCT and Zarcanum" paper (Cypher Stack–audited).
- The Zano C++ reference implementation (`crypto/` and wallet code) — the ground truth for serialization, generators, and signing equations.
- The Cypher Stack audit of d/v-CLSAG.

Every equation, generator, and hash-to-point/domain-separation constant must be lifted from the reference code and reproduced bit-for-bit, then checked against test vectors generated by the Zano node.

## 2. Reference architecture (adopt the Monero model)

The Monero-on-Trezor design is the template because it solves the two problems you also have — a resource-constrained device and a privacy protocol that needs the whole tx built on-device:

- The device is treated as a **cold wallet**. The host wallet knows the blockchain; the device knows the secrets. The transaction is assembled incrementally on the device across many messages.
- The host is **untrusted storage**. Because the device can't hold a large tx in RAM, intermediate state is offloaded to the host, but every offloaded blob is HMAC'd with a unique, index-dependent key so the host cannot tamper, reorder, replay, or reuse it.
- **Range-proof offload (MPC):** for the common ≤2-output case the device does the full Bulletproof(+); for more outputs the heavy computation is offloaded to the host and the device *verifies* it, using a secure multi-party protocol with constant device-side memory.
- A separate **key-image sync** protocol lets a view-only host wallet compute correct balances (otherwise spent outputs look unspent and balances double-count).

Everything above maps 1:1 onto Zano. The deltas are the crypto primitives inside these envelopes and the tx serialization.

## 3. Workstreams / components

### A. Cryptographic primitives — `trezor-crypto` (C), performance-critical
- Reuse: Ed25519 group ops, scalar arithmetic, hash-to-point, base58 (CryptoNote variant), Pedersen commitments.
- Build new: d/v-CLSAG sign/verify; Confidential Assets generators + blinded asset tags + balance proof; Bulletproofs+ adapted to Zano's exact statement; Zarcanum `Q`-element handling (if staking in scope).
- Requirement: constant-time, side-channel-aware, deterministic-nonce. No allocation surprises on-device. Provide a host-buildable Python/Rust binding for differential testing.

### B. Device application — `core/src/apps/zano/`
Mirror `apps/monero/`:
- `protocol/` — high-level stateful signing + key-image sync state machines.
- `xmr/serialize`-equivalent → `zano/serialize` — Zano tx serialization ported from C++.
- `misc/` — address encoding, key derivation, alias display formatting.
- On-device confirmation UI: recipient, amount, asset type/ID, fee — rendered so the user verifies each on the trusted screen. Confidential Assets means you must clearly show *which asset* is moving.

**Implementation-language decision:** the Monero app is MicroPython + C. Trezor now prefers Rust for new features. For a brand-new 2026 app targeting the Safe 7, expect SatoshiLabs to favour a **Rust-forward** app layer (with C/Rust crypto), even though the fastest prototype path is to clone the MicroPython Monero app. Decide this with them early; it materially changes the codebase.

### C. Wire protocol — `common/protob/messages-zano.proto`
Define the message set modeled on `messages-monero.proto`:
- `ZanoGetAddress`, `ZanoGetWatchKey`
- `ZanoTransactionInitRequest/Ack`
- `ZanoTransactionSetInput`, `...SetOutput`, `...AllInputsSet`, `...SignInput`, `...FinalAck`
- `ZanoKeyImageSync{Init,Step,Final}`
Include asset-id fields throughout (Monero's protocol has none). Register message IDs with SatoshiLabs to avoid collisions.

### D. Host-side wallet integration — Zano C++ wallet
This is the analogue of Monero's `src/device_trezor/`. Without it, nothing is usable:
- A `device_trezor`-style backend in the Zano wallet that speaks the protobuf protocol over USB/THP.
- Coordinator logic that drives the multi-round signing, does the offloaded Bulletproof+ work, stores HMAC'd blobs, and reassembles the final tx for broadcast.
- View-only wallet + key-image sync so balances are correct with secrets on-device.
- "Restore wallet from device" flow.
This must be built *with* the Zano core team — it lives in their repo and their release cycle.

### E. Client libraries & UX surface
- `trezorlib` (Python) methods for Zano.
- `trezor-connect` support if any web/dApp flow is desired.
- Trezor Suite: Monero itself isn't in Suite (users drive it from Monero GUI). Realistic v1 for Zano is the same — drive from the Zano desktop wallet, *not* Suite. Suite integration is a later, separate effort.

### F. Coin definition & key derivation
- Add Zano to `common/defs` (coin metadata, slip44, symbol, decimals).
- **Derivation:** Monero uses **SLIP-0010** on Trezor (not the standard BIP-32/39 path) for its extra security requirements — with the known trade-off that the backup may only be restorable to another Trezor, not to arbitrary Zano wallets. Make this derivation decision deliberately and document the recovery story loudly; it's a support-and-UX landmine if left implicit.

## 4. Scope: v1 vs later (be ruthless here)

**v1 (MVP) — "receive and spend native ZANO":**
- Get address / watch key, key-image sync, sign standard transfers of **native ZANO only**.
- ≤2 outputs on-device fast path; N-output offload path.
- Driven from Zano desktop wallet.

**v2+:**
- Confidential Assets (send/receive non-native assets) — significant extra crypto + UI.
- Zarcanum staking on device (extra `Q` handling, PoS mining tx).
- Aliases, auditable-wallet semantics, multisig/escrow.
- Post-**HF6** Gateway Addresses (account-based address type; new address handling) — decide based on whether users need on-device bridging. Note HF6 activates on mainnet at block 3,833,000, ~25–27 Aug 2026, so design around it but don't block v1 on it.
- Trezor Suite integration.

Confidential Assets are tempting to fold into v1 but roughly double the crypto and UI surface. Ship native-only first.

## 5. Testing & verification strategy

- **Differential testing:** a host-side pure implementation (Python/Rust binding of the same primitives) run against the C/Rust device code on identical inputs; every intermediate value must match.
- **Test vectors from the Zano node:** generate real d/v-CLSAG signatures, CA balance proofs, and serialized txs from Zano's own code; assert the device reproduces/validates them.
- **Emulator first:** Trezor Core emulator for the full protocol before touching hardware; measure state size and round-trips (Monero's protocol keeps device state constant-size — hold that bar).
- **Testnet end-to-end:** build → sign on device → broadcast to Zano testnet → confirm acceptance; then double-spend/tamper tests (mutate offloaded blobs and confirm HMAC rejection).
- **Hardware perf budget:** sign time and memory on actual Safe 7 for realistic (inputs, outputs) tuples; fix the worst cases before audit.

---

## 6. Security review & audit (non-negotiable)

- Independent protocol review of the adapted cold-wallet/offload protocol (tamper, replay, reorder, cross-instance interleaving under a malicious host).
- Cryptographic implementation audit of d/v-CLSAG, CA, BP+ on-device (constant-time, nonce handling, domain separation). Cypher Stack is a natural fit given they audited the underlying scheme.
- SatoshiLabs internal security review — a hard gate for any upstream merge.

---

## 7. Upstreaming & governance — the real bottleneck

Two things gate whether ordinary users ever get this:

1. **SatoshiLabs acceptance.** Trezor firmware is open source, but the Safe 7 ships a secure element and signed firmware. Unofficial firmware shows tamper warnings and wipes the device on install; you cannot quietly ship a fork to normal users and keep the device "genuine." So the credible path is a PR merged into official `trezor-firmware`, which means their review, their coding standards (likely Rust), and their release schedule.
2. **Zano core team ownership** of the host-side wallet module, test vectors, and protocol review.

Start both relationships *before* writing production code. Open a design proposal (like the original `monero-trezor-doc`) and socialize it with both teams. Zano already lists hardware-wallet integration on its roadmap, so there is likely appetite and possibly grant funding on their side.

## 8. Team, effort, timeline

**Team (minimum viable):**
- 1–2 applied cryptographers comfortable with CLSAG/Bulletproofs and constant-time C/Rust.
- 1–2 embedded/firmware engineers fluent in Trezor Core (MicroPython/Rust/C).
- 1 host-integration engineer on the Zano C++ wallet.
- Part-time protocol/security reviewer.

**Rough timeline (native-ZANO v1, collaborating with both teams):**
- Q1: design proposal, primitive port + differential tests, protobuf spec, relationships with SatoshiLabs/Zano.
- Q2: device app + serialization; emulator signing for ≤2 outputs.
- Q3: N-output offload/MPC path, key-image sync, host wallet integration, testnet e2e.
- Q4: hardening, perf, external audit, SatoshiLabs review.
- Q5: upstream merge, release, docs, recovery/UX.

Call it **9–18 months** to a merged, audited MVP with a competent team and cooperative counterparties. Solo and/or without SatoshiLabs buy-in: substantially longer and may never reach stock devices. Confidential Assets add another multi-month phase on top.

---

## 9. Key risks

- **Crypto correctness / side channels** — the highest-severity risk; mitigate with differential testing, vectors, and independent audit.
- **Device memory/perf** — CA + BP+ may strain RAM; the MPC offload pattern is your safety valve, but N-asset txs are heavier than Monero's single-asset case.
- **SatoshiLabs won't merge** — mitigate by co-designing early and matching their language/standards; worst case is a fork with degraded UX and no "genuine device" story.
- **Derivation/recovery incompatibility** — the SLIP-0010 trade-off can trap funds in "Trezor-only" recovery; design and document deliberately.
- **Moving target** — HF6 (Aug 2026) changes address types and bridging; pin a protocol version and plan for a follow-up.
- **Scope creep** — CA/staking/aliases/multisig each look small and aren't. Guard the v1 line.

---

## 10. First five concrete steps

1. Pull the Zano reference implementation and the CA/Zarcanum papers; write a one-page crypto delta spec vs. Monero's Trezor app (exact primitives, generators, serialization).
2. Port d/v-CLSAG + CA commitments into a host-buildable library with test vectors generated from the Zano node; get differential tests green off-device.
3. Draft `messages-zano.proto` and a `monero-trezor-doc`-style protocol proposal.
4. Open the conversation with **both** SatoshiLabs and the Zano core team; confirm language (Rust vs MicroPython), message-ID allocation, and host-repo ownership. Ask Zano about grant support.
5. Stand up a Trezor Core emulator branch with an `apps/zano/` skeleton that signs a 1-input/2-output native-ZANO testnet tx end-to-end.

---

*Scope note: this plan targets native-ZANO spend as v1. Confidential Assets, Zarcanum staking, Gateway Addresses, and Trezor Suite are explicitly deferred to keep the first milestone shippable and auditable.*
